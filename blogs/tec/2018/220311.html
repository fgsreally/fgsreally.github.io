改造folio-2019

vuePress-theme-reco fgs    2021 - 2022

改造folio-2019

fgs 2022-03-07 three 未修

# 简述：

有一个开源项目folio-2019，一个法国人做的three.js项目，做过three的应该都知道，毕竟也挂在官网第一行好几年了。有天晚上我在琢磨这个项目的时候，我妈看到了觉得中小学可能会感兴趣买，就让我帮忙做一个

注意：这是一个高度基于matcap的项目，在一些追求真实的场景（B端）可能不太靠谱

# 分析

首先这是一个完整的项目，而非一个开源的库，本身的结构就不适合做二次开发，也没有文档。而且这是3d类的项目，不可能对原有的东西不动的，至少模型啥的肯定是要换的。 此外，该项目开发者更多是一位创意开发者，项目里面的一些写法很糟糕，经常出现a.b.c.d.e.f这样嵌套了很多层的东西，又没有ts，vscode都帮不上忙，three.js配置又复杂，很多时候完全不知道在操作什么属性

# 目标

小规模的更改并无意义，我希望把其改成一个框架，方便二次开发。

# 进一步分析

这个老哥的想法很有意思，他认为前端对于3d表达有两个无法克服的问题，一是光线追踪，前端不能做到这么高水平的实时渲染，二是模型贴图等的传输压力 整个项目中没有调用一个光线类，或者说项目中大部分的工作都在实现“没有光线，但有光线”这个目标，设计上的原理我单独写一章，就不在这写了，总之，这个项目中至少有如下的部分是在为此工作：动态光影，静态光影，视角限制，matcap，effectcompose。 那么这些部分是要尽可能保留的

# 再进一步

项目中主要的乐趣来源是操作小车碰撞，所以当然有三个部分， 一：小车控制模块，二：物理模块，三：音效模块

# 再再进一步

整个项目的代码是非常多的，为了分块，会根据区域进行划分--区域模块 有大量重复的东西，比如墙，砖等等，一个一个写太麻烦，是有重用的模块，也就是 --重用的模型/shader模块 一些基本的，比如资源注入模块，时间模块，页面缩放模块，开场动画模块，调试模块等

# 第一步总结

注：很多模块的耦合度很重，比如物理模块一旦取消，整一个引入模型的模块都无法正常工作了，又比如少引用一个砖块模型，那么砖块重用的模块就没用了 那么模块可以进行如下区分 1基本模块，比如资源加载，这些模块在各种three项目中都有，只是实现稍有不同 2前置模块，是该项目中独有的，且对整个项目影响很大，后续模块必须基于它执行，比如 物理模块，控制模块，小车模块。

3插件模块，也是为后续的模块提供功能，但只要后续模块不调用该功能，项目仍正常运行

4区域模块，前面都是提供功能，这里才是真正的3d编程部分

（只需要把第四步换掉，就相当于二开了） （这里主要麻烦是第二步，因为这些模块本身复杂，对项目影响大，往往又要做一点自定义的东西，存在冲突）

# 面向接口

做这种大东西，先要想调用的形式，目前至少有三种注入 1，置项，也就是参数，主要对应资源等 2，调，比如effectcompose 3，复杂的部分，比如插件，区域

那么一般，只需要准备plugin，可以是内置的，也可以自己编写，然后就可以开始写区域了,就可以随心所欲调用框架及插件的功能了

//区域
export default class CrossroadsSection
{
    constructor(_options)
    {
        // Options
        this.time = _options.time//引入时间，资源，物体添加，
		//砖块，按钮，调试功能
        this.resources = _options.resources
        this.objects = _options.objects
        this.areas = _options.areas
        this.tiles = _options.tiles
        this.debug = _options.debug
        this.x = _options.x
        this.y = _options.y

        // Set up
        this.container = new THREE.Object3D()
        this.container.matrixAutoUpdate = false

        this.setStatic()
        this.setTiles()
    }

    setStatic()
    {
        this.objects.add({//场景中添加一个有物体碰撞的东西
            base: this.resources.items.crossroadsStaticBase.scene,
            collision: this.resources.items.crossroadsStaticCollision.scene,
            floorShadowTexture: this.resources.items.crossroadsStaticFloorShadowTexture,
            offset: new THREE.Vector3(this.x, this.y, 0),
            mass: 0
        })
    }

   
}

如果需要大改动，比如把汽车换成飞机啥的，就要跟改pre中的部分了，当然也可以通过类继承减少负担（额，实际上车是最麻烦的，cannon中的东西写起来非常不舒服，改成其他会简单不少）

# 加一点新东西

实际上，不可能只用项目中原有的东西，必须要加点新的，这又有两种方式， 一，对项目结构调整，支持新功能 二，做新插件 三，更改旧功能

先从二说起，这个插件，主要就是上述的重用模块，这又分为两种，第一种基于模型的，项目中原有的墙，砖都是如此，但这种需要资源模块中必须引入该模型，否则出错，第二种是纯粹用shader写的，这个就没有依赖，想用就用， 插件编写很容易，很多项目中的一些雷达效果什么的可以很容易迁移过来

关于第三点，（其实把物理模块中汽车改成飞机就是一种情况），我举个例子，现在把一个屋子，放在（0，0）的位置，然后他就会自动根据模型名称，引入matcap贴图，根据物理模型，创建碰撞，由于这个过程是自动的，反而有时候很麻烦，比如我可能希望这个东西没有物理碰撞，亦或者我想要贴图不只是matcap，比如，我想在一个地方引爆，然后冲击波蔓延开来，如果按原本自动matcap的话，那么这个冲击波是没办法到这个屋子上的，这里显然需要一个对于shader的hook

第一点我也就做了一个功能（用的是两个rendertarget，更改权重从而实现穿景，比如点击一个按钮，从小车世界“穿越“到教室） 因为是对整个结构的改变，所以不太推荐

# 性能优化

这位老哥通过matcap，让多模型公用同一贴图，减少了贴图传输的压力，从而解决上文中的第二点。但他这里的添加物体模块，还是以建房子为例，每建一个，就要引入一个模型和一个用于物理碰撞的模型，相当于每有一个引入，就要远程多加两个glb,实际上用到的模型可能有数百个，这个传输是不能接受的，这位老哥提供了一种解决方案，把十几栋屋子合并成一个文件，减少传输，但这样的引入会是一个固定的整体，比如我用小车装上去一栋屋子，大概率这十几栋屋子会一起后退

一般来讲，这种模型传输的优化都是往减面思路上走，但这个呢，怎么说呢，反正从设计上不太需要高精模型，所以还是要通过减少传输次数来提高性能，就需要建多个房子时，可以从同一个glb的children中读取自己需要的部分，这需要把object模块的接口改造一下。
